# AHMS Backend: Database Documentation

## Overview

The AHMS backend uses SQLite for development and provides a clear migration path to PostgreSQL for production. The database persists all telemetry ingestion, normalization, health reasoning, and audit trail data.

## Database Setup

**File Location:** `ahms.db` (in project root by default)

**Configuration:** 
- Database directory can be overridden via `DATABASE_DIR` environment variable
- SQL echo logging can be enabled via `SQL_ECHO=true` environment variable

**Initialization:**
```python
from src.store.sqlite import init_db
init_db()  # Creates all tables from ORM models
```

## Database Schema

### Tables Overview

| Table | Purpose | Key Fields |
|-------|---------|-----------|
| `ci_events` | Raw ingested telemetry | id, source, event_type, normalized, system_name |
| `health_reports` | Computed health assessments | id, health_score, primary_issue, reasoning, confidence |
| `remediation_suggestions` | Action recommendations | id, action, rationale, severity, confidence |
| `audit_logs` | Immutable decision trail | id, event_type, decision, rationale, timestamp |

### Table: `ci_events`

Raw telemetry from observability, workflow, or batch systems.

```sql
CREATE TABLE ci_events (
  id TEXT PRIMARY KEY,
  source TEXT NOT NULL,  -- observability, workflow, batch
  source_id TEXT NOT NULL,  -- External system ID
  event_type TEXT NOT NULL,  -- metric, log, trace, business_event
  timestamp DATETIME NOT NULL,  -- When event occurred
  received_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  environment TEXT NOT NULL,  -- ci, staging, production
  
  -- Raw payload
  data JSON NOT NULL,
  
  -- Normalization pipeline
  normalized JSON,  -- Standardized form
  normalization_status TEXT DEFAULT "pending",
  normalization_error TEXT,
  
  -- Tracking
  lineage JSON,
  system_name TEXT,  -- System being monitored
  correlation_id TEXT,  -- Cross-event linkage
  ingestion_attempt INTEGER DEFAULT 1,
  deduplicated TEXT DEFAULT "false"
);

-- Indexes for fast queries
CREATE INDEX idx_ci_events_system ON ci_events(system_name);
CREATE INDEX idx_ci_events_environment ON ci_events(environment);
CREATE INDEX idx_ci_events_timestamp ON ci_events(timestamp);
CREATE INDEX idx_ci_events_correlation ON ci_events(correlation_id);
```

### Table: `health_reports`

Computed health assessments generated by the reasoning engine.

```sql
CREATE TABLE health_reports (
  id TEXT PRIMARY KEY,
  system_name TEXT NOT NULL,
  environment TEXT NOT NULL,  -- ci, staging, production
  
  -- Health score
  status TEXT NOT NULL,  -- healthy, warning, critical, unknown
  health_score REAL NOT NULL,  -- 0.0 - 1.0
  confidence REAL,  -- 0.0 - 1.0
  
  -- Objective findings
  primary_issue TEXT,  -- Main problem detected
  issues JSON,
  
  -- Reasoning chain
  reasoning JSON,  -- {reasoning_chain: [...], evidence_count: N}
  
  -- Suggestions
  suggestions JSON,  -- Array of action strings
  
  -- Correlations and anomalies
  correlated_events JSON,  -- List of event IDs
  anomalies_detected JSON,  -- List of anomaly dicts
  
  -- CI verdict
  ci_verdict TEXT,  -- pass, warn, fail
  verdict_rationale JSON,
  
  -- Governance
  recommended_actions JSON,
  governance_check TEXT,
  
  -- Audit
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  audit_notes JSON
);

-- Indexes
CREATE INDEX idx_health_reports_system ON health_reports(system_name);
CREATE INDEX idx_health_reports_environment ON health_reports(environment);
CREATE INDEX idx_health_reports_created ON health_reports(created_at);
```

### Table: `audit_logs`

Immutable audit trail of all backend decisions.

```sql
CREATE TABLE audit_logs (
  id TEXT PRIMARY KEY,
  event_type TEXT NOT NULL,  -- ingestion, reasoning, decision, action_approved, etc.
  timestamp DATETIME DEFAULT CURRENT_TIMESTAMP,
  
  -- Context
  user_id TEXT,
  system_name TEXT NOT NULL,
  environment TEXT NOT NULL,
  
  -- Event details
  event_data JSON NOT NULL,
  decision TEXT,
  rationale TEXT,
  
  -- Links
  related_event_id TEXT,
  related_report_id TEXT,
  
  -- Immutability marker
  immutable TEXT DEFAULT "true"
);

-- Indexes
CREATE INDEX idx_audit_system ON audit_logs(system_name);
CREATE INDEX idx_audit_event_type ON audit_logs(event_type);
CREATE INDEX idx_audit_timestamp ON audit_logs(timestamp);
```

## Integration with Report Generation

### POST /api/reports/generate Flow

The report generation endpoint integrates with the database as follows:

```
1. Request arrives: 
   POST /api/reports/generate
   {
     "system_name": "my-service",
     "environment": "ci",
     "lookback_minutes": 60
   }

2. Query database for recent CI events:
   SELECT * FROM ci_events 
   WHERE system_name = "my-service" 
   AND environment = "ci"
   ORDER BY timestamp DESC
   LIMIT 100

3. Extract normalized data from events (already processed by normalization pipeline)

4. Run reasoning engine on normalized events:
   - Compute health_score
   - Identify primary_issue
   - Generate reasoning chain
   - Detect anomalies
   - Correlate events

5. Persist health report to database:
   INSERT INTO health_reports 
   (id, system_name, environment, status, health_score, primary_issue, 
    reasoning, confidence, correlated_events, anomalies_detected, ...)
   VALUES (...)

6. Log audit event:
   INSERT INTO audit_logs 
   (event_type="reasoning", system_name=..., event_data=...)
   VALUES (...)

7. Return report to user with:
   - health_score, primary_issue, reasoning chain
   - suggestions for remediation
   - detected anomalies
   - correlated events
```

## Key Database Queries

### Get Recent Events for a System

```python
from src.services.persistence import PersistenceService

events = PersistenceService.get_ci_events(
    db=db,
    system_name="my-service",
    environment="ci",
    limit=100
)

# Returns: List[CIEvent] with normalized data already populated
```

### Get Latest Health Report

```python
report = PersistenceService.get_latest_health_report(
    db=db,
    system_name="my-service",
    environment="ci",
)

# Returns: HealthReport with all reasoning and audit data
```

### Query Audit Trail

```python
logs = PersistenceService.get_audit_logs(
    db=db,
    system_name="my-service",
    limit=50
)

# Returns: List[AuditLog] showing all decisions for this system
```

## Persistence Service Methods

The `PersistenceService` class provides the primary interface to the database:

| Method | Purpose |
|--------|---------|
| `create_ci_event` | Ingest and store raw telemetry |
| `update_ci_event_normalization` | Store normalized form after processing |
| `get_ci_events` | Query recent events by system/environment |
| `create_health_report` | Persist computed health assessment |
| `get_latest_health_report` | Retrieve most recent report for gating/display |
| `create_remediation_suggestion` | Store action recommendations |
| `log_audit_event` | Record governance/reasoning decisions |
| `get_audit_logs` | Query decision trail |

## Migration to PostgreSQL

For production, replace SQLite with PostgreSQL:

1. Install PostgreSQL adapter:
   ```bash
   pip install psycopg2-binary
   ```

2. Update DATABASE_URL in `src/store/sqlite.py`:
   ```python
   DATABASE_URL = "postgresql://user:password@localhost/ahms"
   ```

3. Use Alembic for schema migrations:
   ```bash
   alembic upgrade head
   ```

The ORM models remain identical; only the database backend changes.

```bash

SELECT COUNT(*) FROM ci_events;

SELECT event_type, COUNT(*) FROM audit_logs GROUP BY event_type;
```
